Choosing the right  data structures is key to designing efficient algorithms.

Stack

A stack is a data structure that retrieves data in the reverse of the order in which the data is stored, much like taking the top-most book off of a stack of books. The last element added is the first element retrieved, sometimes called “last-in, first-out” or “LIFO”.

Queue

A foil to the stack, the first element added is the first element retrieved (“first-in, first-out” or “FIFO).  This is exactly what it sounds like - first come, first served, like any traditional line.

The important thing to remember about stacks and queues is that we only ever operate on their “head” or “tail” elements.  Unlike arrays, in stacks and queues we do not have access to elements in the middle.

Arrays

Arrays are data structures that store data of the same type in adjacent, contiguous memory locations.

These memory locations are all the same size, can hold the same amount of data, and can be accessed by an index number (index 0, of course).
It is important to note that array size is fixed when an array is declared, and a certain amount of memory is allocated. There's no guarantee that more memory adjacent to your array will be available for use later, so arrays cannot easily grow.
By using the array index, we have random access to any/ all of the elements in an array.  In fact, if we know the index number, the lookup time in big O notation is a constant.

Associative Arrays (aka Hash Table)

 [Associative arrays are arrays where, instead of just numeric index values, memory locations are also given names.  We would recognize these as the keys in a key/value pair.  The association between a key and a value is sometimes called a  "binding", and "binding" may also be used to refer to the process of creating a new association. - don’t know if this is true/ relevant in this case]

In an associative array, how data is stored is determined by a hash function.

For example, a hash function may choose to store all items beginning with “b” in index 2.
If we have 2 or more items with that index, that will be a “collision”, and we have options for dealing with that difficulty- linear probing and separate chaining.

Using linear probing, if hashing function directs an item (a key) to the same index as a previously stored key, it is assigned the next available area in the table.  This quickly makes the efficiency of the table devolve to O(n).  

Using the separate chaining model, the hash table becomes an array of pointers to linked lists.

When a collision occurs, the key can be inserted in constant time at the head of the appropriate linked list, making the lookup time a much more respectable O(n/k) (with k being the size of the linked list).
